## Session Overview

**Topic:** 面向芯片设计（数字后端/3DIC）的大文件版本管理与团队协作系统
**Goals:** 设计一个能管理EDA工具链大量输入输出大文件、支持团队协作、解决存储爆炸/I/O性能/追溯缺失等痛点的版本管理方案

### 核心约束与痛点

- **存储爆炸：** 常规文件几十MB~几百MB，GDS/OASIS可达几十GB~几百GB；一个项目每天产生TB级新数据
- **I/O性能：** EDA工具对文件读写性能敏感，版本管理不能成为I/O瓶颈
- **协作低效：** 当前靠命名规则手动管理，无分支/合并/锁定机制
- **追溯缺失：** 最终产出物（GDS/PR DB）无法追溯源头、定位引入问题的版本
- **工具链约束：** Git LFS难满足量级需求，Perforce可行但无法采购商业许可

### 关键对标

Perforce Helix Core核心能力：百万级文件管理、增量传输（Delta Transfers）、全局复制与缓存、单一仓库管理、IP生命周期集成

### Session Setup

用户来自芯片设计领域，团队当前通过手动命名规则管理大文件版本。已调研过Git LFS（不满足需求）和Perforce（无法采购商业许可）。希望自研一个类似Perforce的开源方案，专门面向芯片设计场景优化。

## Technique Execution Results

### First Principles Thinking（第一性原理思考）

**交互焦点：** 从芯片设计流程的本质出发，逐层拆解"版本管理"的底层假设

**7条基本事实：**

1. **版本是探索节点，不是时间快照** — 版本标记的是"用这组约束沿这个方向的探索结果"，不是某个时间点的文件状态
2. **输出可从配方重现，但重现代价极高** — 同样输入+参数→基本相同输出，但重跑一次需要几小时到几天
3. **分支有天然淘汰机制** — 探索→评估→保留Top N→淘汰其余，不需要永久保存所有分支
4. **版本图有环** — 下游失败驱动上游变更（如PR时序修不掉→回退RTL修改），与Git的DAG假设根本不同
5. **系统管理的是设计决策历史，文件是副产品** — 包括回退原因等决策信息需要系统管理
6. **追溯的本质是沿版本图回溯决策链** — "为什么走到这里"比"当时文件长什么样"更重要
7. **版本的核心原子单位是"制品集"，不是单个文件** — 输入输出天然配套，单独文件无意义；文件集合级别是流程视角最重要的粒度

**关键突破：**
- "配方vs蛋糕"比喻揭示了存储策略的核心权衡——配方极小但重现代价高，蛋糕即用但存储爆炸
- 版本图有环这一事实直接否定了Git式DAG架构的适用性
- 制品集作为核心原子单位，决定了系统的数据模型设计

**用户创意贡献：**
- 提出"重烤"时间成本，纠正了纯配方策略的片面性
- 明确版本图中的环（回退场景）
- 确定文件集合为最小原子单位，区分流程视角和工程视角的不同需求
- 参考DSO.ai/Cerebrus的设计空间探索策略

### Morphological Analysis（形态学分析）

**交互焦点：** 系统性探索6个设计维度的参数组合

**6维度结论：**

| 维度 | 结论 |
|------|------|
| **A. 存储模型** | CAS + Chunking为底层统一存储引擎（天然支持跨分支去重、O(1)读取），Delta作为传输优化补充，分层策略决定"存什么" |
| **B. 版本图模型** | 底层制品集血缘（表达流程关系）+ 上层文件版本视图（方便工程师查看历史），双层模型结合。制品集引用文件版本，多对多关系 |
| **C. 分支/淘汰策略** | 分阶段生命周期（Exploring→Candidate→Pinned→Golden），手动淘汰+引用计数+策略驱动，具体阈值待实现阶段确定 |
| **D. 协作模型** | 分支隔离 + 验证门禁晋升，不用锁（降低学习成本） |
| **E. I/O架构** | 版本仓库部署在NAS上就地存储，一分支一目录零拷贝切换，版本切换用CAS增量拼接，显式commit不依赖文件监控 |
| **F. 追溯机制** | 血缘查询（向后）+ 影响分析（向前）+ 基线快照（宏观），三个方向覆盖全场景 |

**关键突破：**
- CAS vs Delta的深度对比揭示了CAS更适合作为底层引擎——因为它不依赖版本关系、跨分支天然去重、读取恒定性能
- 双层版本模型（制品集+文件版本）统一了流程视角和文件视角的需求
- NAS原生集成方案——版本仓库就在NAS上，工程师工作方式不变
- 一分支一目录解决了分支切换的拷贝问题

**用户创意贡献：**
- 明确Delta和CAS都是之前考虑过的方案，但之前没想到结合
- 提出分层策略中"大部分版本只存配方，用户标记的版本保存全部"
- 确认NAS环境（类似本地磁盘性能）为基本前提
- 提出reflink不可用，方案需要更好的兼容性
- 确认工程师主动淘汰分支的工作模式
- 提出CI/CD式协作的构想

### Cross-Pollination（跨领域交叉授粉）

**交互焦点：** 从影视/VFX、游戏开发、科学计算三个领域借鉴大文件管理方案

**5个可迁移灵感：**

1. **Work/Publish分离（VFX）** — 发布模型 = 分阶段淘汰的UI表达，工程师天然理解"发布=正式版本"
2. **验证门禁Agent（游戏/Mergebot）** — 每次提交制品集自动触发验证，验证结果决定能否晋升，是CI/CD式协作的具体实现路径
3. **配方即缓存Key（科学计算/DVC+Nextflow）** — hash(输入+参数)判断是否需要重跑，避免重复计算，配方不只是存储策略还是缓存策略
4. **参数+指标+版本绑定（科学计算/DVC）** — 工具参数/PPA指标/制品集版本三位一体，工程师可直接在版本管理系统中对比各分支PPA指标
5. **延迟加载（VFX/OpenUSD Payload）** — 只checkout需要的文件子集，不全量加载

**关键发现：** 跨领域方案基本验证了已有方向，未发现重大新方向。这说明第一性原理和形态学分析已经覆盖了核心设计空间。

## Idea Organization and Prioritization

### Thematic Organization

**Theme 1: 存储引擎 — 怎么存和去重**

_Focus：大文件的物理存储与去重机制_

- **[CAS + Chunking为底层引擎]** — 内容寻址分块，跨分支天然去重，O(1)读取
- **[Delta作为传输补充]** — 已知连续版本间先做delta压缩再切chunk，减少网络传输
- **[分层存储策略]** — 根据版本阶段决定存配方还是存蛋糕
- **[配方即缓存Key]** — hash(输入+参数)匹配时直接复用已有输出，不需重跑工具

_Pattern Insight：存储问题的核心不是"怎么压缩"，而是"什么值得存"——分层策略在前决定存什么，CAS在后决定怎么存，两者正交组合覆盖了存储爆炸的所有场景_

**Theme 2: 数据模型 — 怎么表达版本关系**

_Focus：版本图的抽象结构和核心实体_

- **[版本是探索节点，不是时间快照]** — 打破了Git式线性快照的假设
- **[版本图有环]** — 回退边是芯片设计的刚需，否定了纯DAG架构
- **[制品集为核心原子单位]** — 输入输出天然配套，单独文件无意义
- **[双层模型：制品集血缘 + 文件版本视图]** — 底层流程关系，上层文件身份，多对多关联

_Pattern Insight：数据模型的关键突破是"双中心"——制品集中心表达流程关系（用什么跑了什么得到什么），文件中心表达身份演化（这个文件从v1到v2变了什么），两者通过引用关联。这比Git的单中心（commit）和Perforce的单中心（文件版本）都更贴合芯片设计的复杂性_

**Theme 3: 生命周期管理 — 怎么管理探索和淘汰**

_Focus：分支从创建到淘汰的完整生命周期_

- **[Exploring→Candidate→Pinned→Golden四阶段]** — 自然匹配工程师的探索节奏
- **[手动淘汰 + 引用计数 + 策略驱动]** — 三种淘汰机制覆盖不同场景
- **[Work/Publish分离]** — VFX的发布模型是分阶段策略的天然UI表达
- **[自动降级机制]** — 低阶段自动降级以控制存储空间

_Pattern Insight：生命周期管理的本质是"自由与秩序的平衡"——Exploring阶段给工程师最大自由（尽情探索，只存配方），Golden阶段给项目最大保障（只读冗余），中间阶段是渐进约束。工程师不需要一次性做所有决策，系统自动在背后调整存储策略_

**Theme 4: 协作与集成 — 怎么多人一起工作**

_Focus：团队协作模型和工具链集成_

- **[分支隔离日常开发]** — 工程师在自己分支上自由工作，互不干扰
- **[验证门禁晋升]** — 系统自动跑验证，结果决定能否合并到主线
- **[不用锁]** — 降低工程师学习成本
- **[参数+指标+版本绑定]** — PPA指标直接绑定制品集版本，决策有数据支撑

_Pattern Insight：协作模型的核心转变是"从手动集成到自动流水线"——工程师不再手动拷贝到dataout目录等人审核，而是提交到自己的分支让系统自动验证晋升。这个转变是质的飞跃，验证门禁Agent是关键实现路径_

**Theme 5: I/O与部署 — 怎么保证性能**

_Focus：NAS环境下的存储架构和文件访问性能_

- **[版本仓库部署在NAS上]** — 就地存储，零网络传输
- **[一分支一目录]** — 分支切换=目录切换，零拷贝
- **[CAS增量拼接]** — 版本切换只写变化的chunk
- **[显式commit]** — 不依赖文件监控，工程师主动提交

_Pattern Insight：I/O架构的设计哲学是"零侵入"——工程师的工作方式完全不变（仍然在NAS路径下读写），版本管理在背后透明运行。性能约束通过架构选择而非代码优化来满足_

### Breakthrough Concepts

- **版本图有环** — 对Git/Perforce架构的根本性否定，直接决定了整个系统的数据模型选择
- **制品集+文件双中心模型** — 统一了流程视角和文件视角的多对多关系，现有版本控制工具都没有的设计
- **配方即缓存Key** — 配方不只是存储策略，还是避免重复计算的缓存策略，一石二鸟

| 显式commit工作流 | 高 | 和Git同理，工程师容易理解和接受 |

### 最创新想法
