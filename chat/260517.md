# BMad 头脑风暴会话记录 — 2026-05-17

## 会话主题

**面向芯片设计（数字后端/3DIC）的大文件版本管理与团队协作系统**

## 会话目标

设计一个能管理EDA工具链大量输入输出大文件、支持团队协作、解决存储爆炸/I/O性能/追溯缺失等痛点的版本管理方案

## 核心约束与痛点

- **存储爆炸：** 常规文件几十MB~几百MB，GDS/OASIS可达几十GB~几百GB；一个项目每天产生TB级新数据
- **I/O性能：** EDA工具对文件读写性能敏感，版本管理不能成为I/O瓶颈
- **协作低效：** 当前靠命名规则手动管理（如 xxx.v1.netlist），无分支/合并/锁定机制
- **追溯缺失：** 最终产出物（GDS/PR DB）无法追溯源头、定位引入问题的版本
- **工具链约束：** Git LFS难满足量级需求，Perforce可行但无法采购商业许可

## 关键对标

Perforce Helix Core核心能力：百万级文件管理、增量传输（Delta Transfers）、全局复制与缓存、单一仓库管理、IP生命周期集成

## 用户背景

- 来自芯片设计领域
- 团队当前通过手动命名规则管理大文件版本
- 已调研过Git LFS（不满足需求）和Perforce（无法采购商业许可）
- 希望自研一个类似Perforce的开源方案，专门面向芯片设计场景优化
- 工程师在NAS磁盘上操作，性能类似本地磁盘
- reflink不可用，方案需要更好的兼容性

## 阶段一：第一性原理思考（First Principles Thinking）

### Q1：一个"版本"的本质是什么？

**用户回答要点：**
- 芯片设计过程是一个巨大的设计解空间探索
- 工程师关注spec、输入件和面积/时序/频率/功耗/DRC等约束
- 目标是到达PPA（性能/功耗/面积）的最佳平衡点
- 工程师在多个可能存在最佳解的方向做探索（如不同floorplan方案）
- 版本标记的是过程中输入输出及中间件的状态
- 最终找到一个预期最好的版本来做精细迭代和signoff交付

**核心洞察：** 版本是探索树上的节点，不是线性时间快照

```
        floorplan方案A ──→ 结果A1 ──→ 精细迭代A1' ──→ Golden
       /
起点 ──
       \
        floorplan方案B ──→ 结果B1 (PPA不佳，放弃)
         \
          floorplan方案C ──→ 结果C1 ──→ 精细迭代C1' (时序未收敛，放弃)
```

### Q2：输出是否可以从输入重现？

**用户回答要点：**
- 不完全一致，因为EDA工具算法灵活，功能相同的电路设计上可能存在差异
- 但基本验证（如LVS）可以保证输出在数学定义上一致
- 可以简单认为同样的输入和参数产生的输出基本相同

### Q3：被放弃的分支需要保留吗？

**用户回答要点：**
- 确定走不通的方案版本，一般会因为存储限制删掉不保留
- 评估标准很多（时序、congestion、绕线资源等），工程师综合评估
- 一般感觉比较好的前几个版本会保留
- 中间产物的价值是让工程师回到某个状态点重新尝试
- 参考工具：Synopsys的DSO.ai和Cadence的Cerebrus（设计空间自动探索工具）

### Q4：版本图的结构是什么？

**用户确认：** 整个流程的版本图展开确实是一个DAG图

**但重要补充——版本图有环：**
- 芯片设计流程中是有环的
- 比如在PR过程中发现问题时，要回退到前面某个阶段的版本
- 甚至可能回退到RTL阶段在编码阶段修改

```
RTL v1 → 综合 v1 → PR v1 → 发现时序修不掉
                                  ↓
              ← 回退到RTL修改 ←
                  ↓
RTL v2 → 综合 v2 → PR v2 → ... → Golden
```

**这导出版本图需要两类边：**
- **派生边：** A的输出作为B的输入（正向数据流）
- **回退边：** 因为B的问题，回退到A重新开始（反向因果流）

### Q5："配方vs蛋糕"的时间成本

**用户重要纠正：**
- "配方和蛋糕"的比喻很好，但芯片设计每一步"烘焙"时间极长
- 复杂设计执行一个步骤几个小时起步，很多甚至要几天
- 时间代价必须综合考虑

**由此导出分层存储策略：**

| 版本状态 | 存什么 | 为什么 |
|----------|--------|--------|
| 正在探索 | 只存配方 | 随时可能淘汰，不浪费存储 |
| 评估中（Top N） | 配方 + 压缩缓存 | 可能要用，但还没确定 |
| 确认保留 | 配方 + 完整蛋糕 | 即时可用，不需要重现 |
| Golden/Signoff | 完整蛋糕 + 冗余备份 | 不可丢失，追溯的终点 |

### Q6：回退原因需要系统管理吗？

**用户回答：** 暂定回退的原因等信息是需要系统管理的

### Q7：版本管理的最小原子单位是什么？

**用户回答要点：**
- 两个版本间可能99%二进制内容相同，只部分改变
- 三个场景都存在（整个文件级别、文件内部结构级别、文件集合级别）
- 但从芯片设计流程/流水线角度看，**文件集合级别最重要**
- 输入和输出都是配套的：输入的脚本/网表/sdc/techlib/IP等，输出的网表/PR DB文件/rpt报告/工具日志等
- 其它两个场景更多是工程师设计中会用到

## 第一性原理拆解——7条基本事实

1. **版本是探索节点，不是时间快照** — 版本标记的是"用这组约束沿这个方向的探索结果"
2. **输出可从配方重现，但重现代价极高** — 几小时到几天，配方策略必须考虑时间成本
3. **分支有天然淘汰机制** — 探索→评估→保留Top N→淘汰其余，不需永久保存
4. **版本图有环** — 下游失败驱动上游变更，形成回退边（与Git的DAG假设根本不同）
5. **系统管理的是设计决策历史，文件是副产品** — 包括回退原因
6. **追溯的本质是沿版本图回溯决策链** — "为什么走到这里"比"当时文件长什么样"更重要
7. **版本的核心原子单位是"制品集"，不是单个文件** — 输入输出天然配套，单独文件无意义
