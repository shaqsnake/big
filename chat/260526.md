## PRD 编辑 — Step E-3：Edit & Update

逐项执行变更：

### 变更1 (Critical): NFR模板扩展

全部5个NFR子表（Performance/Security/Reliability/Scalability/Integration）从三列扩展为四列（NFR / 指标 / 测量方法 / 依据），20个NFR均补充了具体测量协议。

示例：
- NFR1测量方法：`dd/fio基准测试 + EDA工作负载回放（PT DB读取、DEF加载），文件级CAS直接读取场景，数据量≥10GB`
- NFR6测量方法：`10个客户端同时对同一分支执行commit，验证无数据损坏和丢失，连续测试100轮`

### 变更2 (High): FR格式修正

- FR10: "分支以独立目录形态存在于NAS上" → "工程师可以在NAS上以独立目录访问各分支"
- FR11: "工程师切换分支时，文件路径结构保持不变" → "工程师可以在切换分支时保持文件路径结构不变"
- FR20: "处于Exploring阶段的制品集只保留配方数据" → "系统可以在Exploring阶段只保留制品集的配方数据而不保留输出文件"
- FR22: "Golden阶段的制品集设置为只读" → "系统可以将Golden阶段的制品集设置为只读，禁止修改或删除"
- FR23: "Golden阶段的制品集存储多副本冗余备份" → "系统可以为Golden阶段的制品集存储多副本冗余备份"

### 变更3 (High): FR18重分类

FR18 "血缘追溯的结果可以在30秒内返回" 从FR章节删除。该指标已在NFR3中完整覆盖（血缘追溯响应时间 < 30秒），无需重复。

### 变更4 (High): FR阶段标签

全部32个FR标注MVP/Growth标签：

- **MVP (20个)**: FR1-15, FR19-21, FR24, FR26, FR32-33
- **Growth (12个)**: FR16-17, FR22-23, FR25, FR27-31

分配依据：对齐Product Scope章节的MVP/Growth/Vision划分。

### 变更5 (Medium): FR33可追溯性

在Journey 1的情感转折段落末尾补充：
> 更让他惊喜的是，当项目后期文件数增长到百万级别时，他可以用 `big checkout --subset` 只加载当前需要的输出文件，而不必等待全量checkout——BIG在大规模场景下同样高效。

Journey 1能力需求列表新增："大规模项目的选择性checkout"。

### 变更6 (Medium): 成功标准旅程支撑

- **团队采纳率**：Journey 1开头补充"这种痛苦正是团队愿意迁移到BIG的根本原因"
- **I/O性能**：Journey 2情感转折补充顶层修时序场景中CAS直读的性能体验叙事

### 变更7 (Medium): Code Examples章节

在Developer Tool Specific Requirements > Documentation后新增Code Examples子章节，包含三类实操示例：
- Python API调用示例（`import big; big.commit(...)`）
- Pipeline定义文件示例（TOML格式的rtl-to-backend流水线）
- 常见CLI工作流脚本示例

### 变更8 (Medium): Migration Guide扩充

在Migration & Onboarding章节扩充：
- 目录结构映射表（现有规范 → BIG目录）
- 命令对照表（手动命名操作 → BIG命令）
- 双轨运行期操作规范及过渡期结束标志
