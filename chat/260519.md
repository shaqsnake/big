## 阶段二：形态学分析（续）

### 维度A：存储模型

**Q1：存储模型的候选方案中，你对哪些有倾向？**

**用户回答：** 2-6都需要考虑，其中Delta（增量差异）和CAS + Chunking（内容寻址分块）是之前就想过的方案，但之前没想过把两者结合。

**AI对比分析：Delta vs CAS + Chunking**

| | Delta | CAS + Chunking |
|---|---|---|
| 实现复杂度 | 中 | 中高 |
| 读取性能 | 链式回溯，越老越慢 | O(1)拼接，恒定 |
| 跨分支去重 | 不支持（需要知道比较哪两个版本） | 天然支持（不管版本关系） |
| 依赖版本关系 | 必须知道谁派生自谁 | 完全无关 |
| 元数据开销 | 小 | 较大（每块一条索引） |

**核心洞察：** Delta和CAS是两种截然不同的去重思路——Delta是"版本间去重"（必须知道比较哪两个版本），CAS是"内容去重"（不管版本关系，天然去重）。对芯片设计意味着：工程师做了floorplan方案A和方案B，GDS可能90%相同但这两个版本没有派生关系——Delta无法处理（A和B不是A→B的增量），CAS天然处理（相同块自动去重）。因此CAS更适合作为底层统一存储引擎，Delta作为传输优化补充。

**Q2：分层策略中，什么条件下从"只存配方"升级到"存配方+蛋糕"？**

**用户回答：** 分层策略是好想法，因为大部分大版本实际上都只需要存配方，只有用户标记的版本可以保存全部的文件。

**维度A结论：** CAS + Chunking为底层统一存储引擎，Delta作为传输优化补充，分层策略决定存什么

### 维度B：版本图模型

**Q3：你对版本图模型有什么倾向？**

**用户回答：** 当前觉得是模型1（显式DAG + 回退标注），但模型4（制品集血缘）也感觉可能符合，希望详细解释模型4。

**模型4（制品集血缘）详细解释：**

核心思想：反转视角——不以"版本"为中心，以"制品集"为中心。

- 模型1中：节点是"版本"，边是"版本间关系"，文件挂在版本上
- 模型4中：节点是"制品集"，边是"输入→输出的依赖关系"，版本是制品集的属性

回退在模型4中的表达：不是"版本v2回退到版本v1"，而是"制品集B因为时序问题催生了新的制品集D"。

```
制品集A（PR输入）──跑PR──→ 制品集B（PR输出，时序未收敛）
                                    │
                                    │ 回退边：因为时序问题
                                    ↓
制品集D（修改后的RTL）──跑综合──→ 制品集E（新综合输出）──跑PR──→ 制品集F（新PR输出）
```

**模型4 vs 模型1：**

| | 模型1 | 模型4 |
|---|---|---|
| 中心实体 | 版本（Version） | 制品集（Artifact Set） |
| 边的含义 | "版本v2从版本v1派生" | "制品集B的输出是制品集C的输入" |
| 回退的表达 | 版本节点上的特殊标注 | 新制品集 + 因果链 |
| 追溯方式 | 沿版本DAG回溯 | 沿输入输出依赖链回溯 |

**Q4：模型4和模型1的关系——是选一个还是结合？**

**用户关键洞察：**

> 从流程的角度来抽象每个步骤就是制品集的概念，输入/执行工具/输出/数据采集等，但确实版本更应该贴合文件的概念，因为存在一个完全相同的文件，比如netlist v1可以在不同的工具参数下产生不同的输出制品集，所以两个概念应该要结合起来一起设计。

**核心洞察：** 制品集维度表达的是"流程"（同一输入，不同参数，产生不同输出），文件版本维度表达的是"身份"（netlist v1就是netlist v1，不管它被用在哪个制品集里）。两者是多对多关系：

```
netlist v1 ──┬──→ 制品集A（PR，参数集α）──→ GDS A
              │
              └──→ 制品集B（PR，参数集β）──→ GDS B

netlist v2 ────→ 制品集C（PR，参数集α）──→ GDS C
```

**数据模型核心 = 两层 + 关联：**
- **文件层：** 每个文件有独立版本线，存储用CAS + Chunking
- **制品集层：** 每个制品集引用一组特定版本的文件 + 工具参数 + 输出文件
- **关联层：** 制品集引用文件版本，一个文件版本可被多个制品集引用

**维度B结论：** 底层制品集血缘 + 上层文件版本视图，双层模型结合

### 维度C：分支/淘汰策略

**Q5：工程师淘汰分支的方式更接近哪种？是主动决定"这个不要了"，还是被动地"忘了删就一直占着空间"？**

**用户回答：** 一般情况下都是主动决定的，1（手动淘汰）、3（引用计数淘汰）、4（策略驱动淘汰）都是存在的场景，对5（分阶段淘汰）需要详细解释。

**分阶段淘汰详解：**

核心思想：一个探索分支的生命周期不是"生"和"死"两个状态，而是像产品一样有多个阶段，每个阶段自动匹配不同的存储和保留策略。

| 阶段 | 存什么 | 保留策略 | 触发升级 |
|------|--------|----------|----------|
| Exploring | 只存配方（KB级） | 可配置最长时间 | 工程师标记"有希望" |
| Candidate | 配方 + 完整产出物 | 超出Top N自动降回Exploring | 工程师标记"选定这个方向" |
| Pinned | 配方 + 完整产出物 + 元数据 | 不自动淘汰，只能手动降级 | 进入signoff流程 |
| Golden | 完整产出物 + 冗余备份 | 永不淘汰，只读 | 终态 |

自动降级机制：
- Golden → 永不降级
- Pinned → 永不自动降级（只能手动）
- Candidate → 超出Top N自动降回Exploring
- Exploring → 超期自动删除配方

和1、3、4的关系：
- 1（手动淘汰）= 工程师手动将某分支从任何阶段直接降到"删除"
- 3（引用计数淘汰）= 没有下游引用的候选制品集，降级提示
- 4（策略驱动）= Top N保留就是策略，每阶段都可以定义规则
- 5（分阶段）= 把1、3、4组织成有方向的生命周期流程，自动匹配存储策略

最大好处：它把"存不存"和"存什么"统一了——阶段1只存配方所以几乎不占空间，工程师可以尽情探索不用操心存储；只有升级到阶段2以上才会占用大量空间，而这时是工程师主动决策的结果。

**Q6：Top N是一个用户可以设置成Candidate状态的数量吗？**

**用户讨论：** Top N有三种理解方式——1）硬上限；2）软提醒+手动确认；3）按"探索上下文"分组（不同设计任务有独立的Top N，因为评估维度不同）。

**用户结论：** 基本匹配，但Top N具体配置还不清楚——应该怎么配置、配置多少、还是完全由用户来定，留到实现阶段再讨论。

**维度C结论：** 分阶段生命周期（Exploring→Candidate→Pinned→Golden），手动淘汰+引用计数+策略驱动，具体阈值待定

### 维度D：协作模型

**Q7：你们团队协作冲突主要发生在什么场景？**

**用户回答：** 当前团队工作方式是每个工程师在自己权限的专用目录下做模块设计，实际上就类似分支隔离的形式；到signoff阶段时统一输出到项目指定dataout目录，按约定的规范来管理，每天都要做这个动作。但不确定这种协作方式是否高效，如果引入了分支和版本管理，是不是能借鉴软件开发中CI/CD流水线的协作方式。

**软件CI/CD映射到芯片设计：**

| 软件开发 | 芯片设计 |
|---------|---------|
| feature分支开发→merge到main | 模块分支开发→集成到项目主线 |
| push代码→自动构建+测试 | 提交制品集→自动跑验证（LVS/DRC/STA） |
| 测试通过→自动部署 | 验证通过→自动晋升到下一阶段 |
| 每天多次merge | 每天一次dataout汇总 |

关键差异：芯片"构建"代价极高（DRC/LVS/STA要几十分钟到几小时），需要分层验证：

```
轻量检查（秒级）：文件完整性、格式校验、命名规范
  ↓ 通过
增量验证（分钟级）：只跑受影响模块的LVS/DRC
  ↓ 通过
全量验证（小时级）：每日定时跑完整signoff验证
  ↓ 通过
晋升到下一阶段
```

协作模型核心变化：
- 现在：工程师手动拷贝到dataout → 人检查 → 人决定能不能进下一阶段
- 之后：工程师提交到自己的分支 → 系统自动验证 → 验证通过自动合并到项目主线

**Q8：你觉得引入锁来协调是否必要？**

**用户回答：** 不考虑锁，因为这也会给工程师带来学习和使用的成本。

**维度D结论：** 分支隔离 + 验证门禁晋升，不用锁

### 维度E：I/O架构

**Q9：你们现在工程师的工作环境是怎样的？文件都存在本地磁盘还是共享存储？EDA工具读写路径是本地还是网络？**

**用户回答：** 当前工程师是在NAS磁盘上操作各种文件，性能类似在本地磁盘读取，这也是项目最基本的要——EDA工具读写不能成为性能瓶颈。1（本地代理缓存）、3（工作区快照）、4（混合模式）应该都是可考虑的方向。

**AI提出方案6：NAS原生集成**

核心思路：工程师工作方式完全不变（仍然在NAS路径下读写文件），版本管理在后台运行（commit是显式动作，checkout是切换视图），版本仓库就是NAS的一个"影子"。

好处：I/O零损失、工程师学习成本极低（只需学"提交版本"和"切换版本"两个动作）、不需要移动大文件。

**Q10：NAS原生集成有几个问题——1）NAS文件监控怎么做？2）怎么和CAS/分支结合？3）checkout切换时具体怎么做？**

**用户提出三个问题：**

1. **NAS文件监控** — 之前尝试过，很难做到，特别是NAS用的网络协议很多原生inotify不支持
2. **保持现有存储模式** — 怎么和前面的分块/分支等技术结合
3. **checkout切换** — 感觉还是会有很多拷贝和替换的动作

**AI逐个解答：**

**问题1：** 放弃自动监控，改为显式commit。Git也是显式commit，不需要实时监控。commit时系统扫描工作区，和上次commit做比对，识别变更。这样NAS监控问题就不存在了。

**问题2：** 版本仓库应该就在NAS上，而不是独立的远程服务器：

```
NAS磁盘
├── /project-x/workspace/engineer-A/   ← 工作目录，工程师直接操作
├── /project-x/workspace/engineer-B/   ← 工作目录
└── /project-x/.big-repo/              ← 版本仓库（同一NAS上）
    ├── objects/                        ← CAS分块存储
    ├── metadata/                       ← 制品集/分支/版本图元数据
    └── staging/                        ← 临时暂存区
```

commit时文件已在NAS上，分块后存入objects，不需要跨网络传输。分支是元数据概念，和文件物理位置无关。

**问题3：** 最初提出reflink（ZFS/Btrfs零拷贝）方案。

**用户确认：reflink不行，方案暂时不考虑reflink，这也让最终系统的兼容性更好。**

**替代方案：一分支一目录**

```
/project-x/
├── .big-repo/                          ← 版本仓库
└── workspaces/
    ├── engineer-A/main/                ← 主分支工作区
    ├── engineer-A/explore-fp1/         ← 探索分支工作区
    ├── engineer-A/explore-fp2/         ← 探索分支工作区
    └── engineer-B/main/                ← 主分支工作区
```

- 切换分支 = 切换目录，零拷贝
- 同分支内版本切换用CAS增量拼接：对比两个版本的chunk列表，相同chunk不动，只重写变化部分

**维度E结论：** 版本仓库部署在NAS上，一分支一目录零拷贝切换，CAS增量拼接，显式commit

### 维度F：追溯机制

**Q11：在实际工作中，最常用的追溯场景是哪些？**

**用户回答：** 1（血缘图查询）、2（影响分析）、5（基线快照）应该都是常用的场景，主要是根据版本来回溯定位问题。

**三个追溯方向的关系：**

1和2是微观追溯（沿单个文件或制品集的依赖链走），5是宏观追溯（看整个项目在某个时间点的全貌）。两者互补：先用基线快照定位"问题出现在哪两次交付之间"，再用血缘/影响分析定位具体是哪个文件的哪个版本引入了问题。

```
                    ┌── 影响分析（向前）──→ "改了这个netlist影响了什么"
                    │
某文件版本 ──┤
                    │
                    └── 血缘查询（向后）──→ "这个GDS是怎么来的"

                    ┌── 基线快照 ──→ "tapeout v2时整个项目是什么状态"
项目 ────────┤
```

**维度F结论：** 血缘查询 + 影响分析 + 基线快照，三个方向覆盖全场景

## 阶段三：跨领域交叉授粉（Cross-Pollination）

**Q12：你想看哪些领域的跨界灵感？**

**用户回答：** 三个领域都想看——影视/VFX制作、游戏开发、科学计算/基因组学。

### 领域1：影视/VFX制作

**相似痛点：** 渲染帧几百MB/帧，一个镜头上千帧，项目PB级；多团队（建模、材质、灯光、合成）协作；需要追溯"最终画面用了哪个版本的灯光和材质"。

**核心模式：Work/Publish分离**

```
/artists/alice/
  work/
    character_model/           ← 工作区，随意修改
      v001/  v002/  v003/     ← 本地版本，自动编号
  publish/
    character_model/
      v002/                    ← 发布版本，经审核后从work提升
        model.usd
        textures/
```

关键设计：
- Work区 = 自由探索，自动递增版本号
- Publish = 显式提交，经审核后提升为正式版本
- 下游只引用Published版本——灯光师只能引用已发布的模型版本
- OpenUSD的Payload延迟加载——几十GB的场景不需要全部加载，按需加载需要的部分

**对芯片设计的启发：**

| VFX | 芯片设计 |
|-----|---------|
| Work区 | 工程师的NAS探索目录 |
| Publish | 通过验证门禁晋升到Candidate/Pinned |
| 下游只引用Published | 模块集成只引用已验证的制品集 |
| Payload延迟加载 | 只checkout需要的文件，不全量加载 |

**最有价值的灵感：** Work/Publish分离就是分阶段淘汰策略的天然UI表达——工程师在work区自由探索（Exploring），publish就是升级阶段。

### 领域2：游戏开发（Plastic SCM）

**核心设计：**
- 分支是容器（不是指针），有自己的生命周期和元数据
- Mergebot：服务器端agent，检测分支就绪后自动在CI上测试，测试通过才合并到主线——这就是软件版的验证门禁
- Gluon：给非程序员（美术师）用的简化界面，只有"更新"和"提交"
- 锁定用于二进制文件，但和分支模型正交，不阻塞工作流

**对芯片设计的启发：**

| 游戏开发 | 芯片设计 |
|---------|---------|
| 分支=容器 | 模块分支=设计任务容器，包含完整上下文 |
| Mergebot | 验证门禁agent，自动跑LVS/DRC/STA |
| Gluon简化界面 | 给不需要理解版本管理的工程师提供简化CLI |

**最有价值的灵感：** Mergebot/验证门禁agent不只是一个概念，而是一个可实现的系统组件——每次工程师提交制品集，agent自动触发验证，验证结果决定能否晋升。

### 领域3：科学计算（DVC + Nextflow）

**DVC核心设计：**
- 元文件代理：不存大文件到Git，只存.dvc元文件（包含MD5哈希），Git管理元文件版本，DVC管理实际数据
- 缓存层：本地.dvc/cache/按哈希寻址存储，相同内容只存一份（本质就是CAS）
- Pipeline DAG：dvc.yaml定义流水线阶段，每个阶段有deps（输入）和outs（输出）
- 可重现性：hash(输入+参数)判断是否需要重跑，跳过未变化的阶段
- 参数+指标+数据版本三位一体

**Nextflow核心设计：**
- 输入hash缓存：每个task的缓存key = hash(输入文件 + 参数 + 脚本 + 环境)，相同key直接返回缓存结果
- 数据溯源：每个输出自动记录是从哪个输入+参数产生的

**对芯片设计的启发：**

| 科学计算 | 芯片设计 |
|---------|---------|
| .dvc元文件 | 制品集的配方（输入文件哈希+工具参数+环境） |
| DVC cache (CAS) | CAS + Chunking存储引擎 |
| Pipeline DAG | 工具链DAG + 回退边 |
| hash(输入+参数)=缓存key | 输出可重现的判定依据 |
| dvc repro跳过未变阶段 | 上游没变下游可跳过重跑 |
| params+metrics+data | 工具参数+PPA指标+制品集版本 |

**最有价值的灵感（两个）：**

灵感A：配方就是缓存Key。DVC用hash(输入+参数)判断是否需要重跑，Nextflow用同样机制避免重复计算。这和"配方优先"策略完美呼应——配方不只是存储策略，还是缓存策略。如果系统发现某个制品集的输入hash和参数完全匹配之前跑过的某个制品集，就可以直接引用之前的输出，不需要重跑。

灵感B：参数+指标+版本绑定。在芯片设计中，一组工具参数跑出结果后，PPA指标（时序余量、面积、功耗）应该和制品集版本绑定记录。这样工程师在选择"保留哪个候选"时，不用打开报告看，直接在版本管理系统中就能对比各分支的PPA指标。

### 跨领域5个关键灵感汇总

1. **Work/Publish分离（VFX）** — 发布模型 = 分阶段淘汰的UI表达
2. **验证门禁Agent（游戏）** — 每次提交制品集自动触发验证，结果决定能否晋升
3. **配方即缓存Key（科学计算）** — hash(输入+参数)避免重复跑工具
4. **参数+指标+版本绑定（科学计算）** — PPA指标直接绑定制品集版本
5. **延迟加载（VFX）** — 只checkout需要的文件子集
