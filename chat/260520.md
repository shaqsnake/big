## 调研成果整理

三个调研任务全部完成后，整理为 `_bmad-output/planning-artifacts/research-technical-architecture.md`，涵盖：

### 一、Perforce Helix Core 架构

- 双层存储：元数据（B-tree键值库 db.*） + 文件内容（gzip压缩档案）
- 文本文件行级增量（reverse delta），二进制文件默认完整副本存储
- Lazy Copy分支：创建分支时不复制文件内容，元数据引用同一档案位置，修改时才copy-on-write
- 文件锁定：`+l`排他锁、`+L`延迟锁
- Commit-Edge分布式架构：Commit Server权威+Edge Server本地缓存+Proxy文件缓存
- Stream模型：mainline/development/release/virtual/task，支持isolate/import路径规则

### 二、DVC 架构

- MD5内容寻址缓存，两级目录结构（哈希前2字符为子目录）
- `.dir`条目：目录类型的JSON元数据文件
- .dvc文件作为大文件的Git追踪代理
- dvc.yaml定义Pipeline DAG（deps/outs/params/metrics）
- dvc.lock锁定精确哈希值，确保可复现
- Run Cache：stage签名 = hash(deps的MD5 + cmd + params)，命中则跳过执行

### 三、Git LFS 架构

- 指针文件替代大文件，SHA-256寻址
- Batch API合并请求，S3预签名URL传输
- 局限：整文件级存储，无文件内部去重，修改1字节也全量重新存储

### 四、CAS/CDC 分块技术

- FastCDC：Gear Hash（每byte 1次查表+1次移位+1次加法），比Rabin快3-5倍
- 归一化分块（两级切割）：块大小分布更集中
- EDA文件推荐参数：min=64KB, avg=256KB-512KB, max=4MB
- Restic Pack格式：多chunk顺序写入+末尾Header记录偏移
- Borg引用计数：每个chunk维护引用计数，归档删除时GC回收

### 五、制品管理系统

- Artifactory：Checksum-Based Storage，SHA-1/SHA-256/MD5三重校验和，相同文件只存一份
- Nexus：Component/Asset双层模型，Blob Store存储二进制
- 共同局限：**无文件内部块级去重**

### 六、综合对比

| 维度 | Git LFS | Perforce | DVC | BIG方案 |
|------|---------|----------|-----|---------|
| 存储粒度 | 整文件 | 整文件(二进制) | 整文件 | CDC分块 |
| 去重方式 | 文件级SHA-256 | Lazy Copy引用 | 文件级MD5 | 块级内容去重 |
| 增量修改效率 | 极低 | 低(全量) | 不支持 | 高(只存变化块) |

**BIG差异化优势：** CDC块级去重、双中心数据模型、配方即缓存Key、分阶段生命周期

## PRD 创建 — Step 2b：产品愿景发现

### Q1：什么会让工程师说"这正是我需要的"？

**用户回答要点：**
1. 解决芯片数字后端设计中版本管理和团队协作的痛点——芯片设计越来越复杂（特别是3DIC），没有清晰高效的版本管理系统，很难支撑后续项目迭代交付
2. 清晰的版本和数据血缘图能帮助工程师快速定位引入问题的来源版本，也能通过分支管理高效管理尝试探索
3. 引入流水线能让整个设计团队交付标准化，保障交付质量

### Q2：当前追溯问题有多严重？

**用户回答：**
- 如果工程师本身管理混乱、命名不规范，很难追溯
- 即使规范管理，也可能执行工具时选错文件，或因某些原因改动了文件但没注意
- 定位一个问题版本至少半天以上，除非版本很少或自己有清晰记录
- 有了BIG后，定位版本应该是秒级，至多分钟级

**AI追问：** "选错文件"意味着BIG不只是记录版本，还应该在提交制品集时校验输入文件是否正确和完整？

**用户确认：** 正确。

### Q3：流水线的具体形态？

**AI解释自动化验证 vs 自动化晋升的区别：**

| | 自动化验证 | 自动化晋升 |
|---|----------|----------|
| 系统角色 | 质检员（只报告结果） | 流水线（报告+执行） |
| 验证通过后 | 工程师手动决定下一步 | 系统按规则自动推进 |
| 好处 | 简单，工程师完全控制 | 减少人工操作，节奏更快 |
| 风险 | 工程师可能忘了或拖延 | 规则设错可能导致错误晋升 |

**用户回答：** 倾向自动化晋升的流水线，但具体场景中是否自动化做晋升还需要讨论——一些情况下是用户人工决定的。流水线更多是把一些必要执行的步骤规范定义好，让用户在某些阶段触发启动，自动完成一系列执行。

**AI总结流水线本质：** 自动化的是执行过程，决策权仍在工程师手中。

```
工程师完成探索 → 手动标记"准备提交" → 触发流水线
  → 自动校验输入文件完整性
  → 自动跑LVS/DRC
  → 自动采集PPA指标
  → 全部通过 → 等待工程师确认晋升到下一阶段
  → 有问题 → 阻塞并通知
```

**用户提供经典流水线示例：**
RTL提交 → 综合 → LVS验证 → 前端出口检查 → 后端入口检查 → 发布到交付制品系统

### Q4：为什么是现在？

**用户回答：** 复杂度到了临界点，且无法采购外面成熟的产品。

## 愿景发现总结

**愿景：** 为芯片数字后端设计（特别是3DIC等复杂流程）提供一个大文件版本管理与团队协作系统，让工程师从手动命名规则的混乱中解放出来，通过制品集血缘图实现秒级问题追溯，通过流水线实现规范化交付，最终支撑越来越复杂的芯片设计项目的迭代和交付。

**核心差异化：**
1. 制品集+文件双中心模型 — 不像Git（commit中心）也不像Perforce（文件版本中心），BIG理解芯片设计的本质是"一组配套的输入输出+工具参数"
2. CAS块级去重 — 比Perforce/Git LFS的整文件存储节省1-2个数量级空间
3. 配方即缓存Key — 不只存数据，还能避免重复计算
4. 流水线驱动交付 — 不只是版本管理工具，还是规范化交付平台

**核心洞察：** 系统管理的是设计决策历史，文件只是副产品。版本的原子单位是制品集，不是单个文件。版本图有环。

**为什么现在：** 设计复杂度到达临界点，且无法采购商业产品（Perforce），必须自研。
