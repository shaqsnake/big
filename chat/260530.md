| 方案 | 版本 | NAS 直读 | MVP 适配 | Growth 可演进 | PRD 合规 |

| A. 自建文件级 CAS | — | **1.0x** | **完美** | 是 | **完全** |

| C. Dulwich | 1.2.5 | 1.5-3x | 低 | 否 | 违反(SHA-1) |
| D. fastcdc+Pack | 1.7.0 | 1.0-1.5x | 不推荐MVP | — | 完全 |
| E. LakeFS | 1.81.1 | 延迟 | 低 | 否 | 违反 |
| F. B2SDK | 2.12.0 | 延迟 | 中低 | 否 | 违反(SHA-1) |

**推荐路径：** MVP 方案 A（自建文件级 CAS + 统一依赖模型）→ Growth 叠加方案 D（fastcdc 块级冷存储）

**统一依赖模型核心设计：**
```

  dependencies: [FileRef],   # 输入文件 + 参数文件，统一 CAS 寻址
  outputs: [FileRef]         # 产出文件
}
FileRef { path, cas_hash, semantic_role("input"|"param"|"config"|"output"), size }
recipe_hash = SHA-256(sorted((path, cas_hash) for all dependencies))
```

## 架构设计 — Step 4：核心架构决策

### SQLite 作为 DAG 存储的深度论证（用户质疑后深入分析）

用户提出："DAG 等相关状态用 SQLite 存是个好方案吗？"——触发全面研究。

**行业验证：**
- **Fossil SCM**（SQLite 之父 Richard Hipp 创建）：15+ 年完全基于 SQLite 存储 VCS 元数据，邻接表 + 递归 CTE，不用闭包表
- **Git commit-graph 反思**：Git 花多年加二进制索引文件，本质在重建数据库能力——SQLite 天然具备

**递归 CTE 实测性能（100K 节点，磁盘数据库）：**

| 操作 | 耗时 | 对应 NFR |
|------|------|---------|
| 血缘追溯 depth 30 | **0.3ms** | NFR3: <30s（超标 10 万倍） |
| 正向影响分析 depth 15 | **0.1ms** | FR17 |
| 配方哈希缓存查找 | **<0.01ms** | FR7 |
| 单次 commit 写入 | **1.6ms** | NFR4 |

**图数据库临界点分析：** 1-5 跳 SQLite 可竞争，6+ 跳图数据库才领先。BIG 的深窄 DAG 形态天然适合递归 CTE。

**Python 集成：** `sqlite3` 标准库模块，零安装零配置零依赖。

### 核心表结构设计

借鉴 Fossil 的 Schema1（不可变核心）/ Schema2（可重建元数据）分层：

**Schema1 — artifact_set, plink, file_ref, dependency, output**
**Schema2 — branch, workspace, audit_log, leaf**

关键决策：整数 RID 外键、is_derived 边类型区分、workspace 表替代文件方案、leaf 缓存表。

### 权限模型：类 Linux 文件权限（5 概念对齐 NFR10）

### CAS 写入完整性：流式 SHA-256 → 临时写入 → 回读校验 → 原子 rename

### 三层级通信架构

```
CLI → core（进程内函数调用）
SDK → core（Python import 直接调用）
GUI → core（FastAPI localhost HTTP API）
```

### 前端架构：Pinia + TanStack Query（Vue Query）

### 部署产物：Python wheel（CLI+SDK）+ Electron 安装包（GUI）

## 架构设计 — Step 5：实现模式与一致性规则

识别 **28 个冲突点**，定义 5 类一致性模式：

### 命名模式
- 数据库：snake_case 单数实体名，`_at` 时间后缀，`is_` 布尔前缀
- API URL：kebab-case 复数名词，查询参数 snake_case
- Python：PascalCase 类、snake_case 函数/变量
- Vue：PascalCase 组件、`use` 前缀组合式函数
- CSS：BEM `block__element--modifier`

### 结构模式
- core 是唯一真相源，CLI/SDK/API 不可包含业务逻辑
- CLI 命令文件命名 `{command}_cmd.py`
- GUI 独立项目 `big-gui/`

### 格式模式
- API 响应：`{data, meta}` / `{error: {code, message, detail, suggestion}}`
- JSON 字段 snake_case，路径 POSIX 相对路径，哈希小写十六进制

### 通信模式
- 事件名 `domain.action` 点分格式，载荷 `{type, timestamp, payload}` 三字段固定结构
- Pinia store 命名 `use{Domain}Store`，仅存 UI 状态不缓存 API 数据

### 流程模式
- 异常层次：BigError → ValidationError / StateError / PermissionError / NotFoundError / StorageError
- CLI 捕获异常格式化输出，SDK 直接传播，API 映射 HTTP 状态码
- 加载态：CLI 进度条，GUI 骨架屏 + Progress 组件

### 执行准则
- 禁止 core 外直接写 SQL（除 metadata 模块）
- 强制错误消息含 message + detail + suggestion
- GUI 禁止直接 fetch，必须用 TanStack Query
